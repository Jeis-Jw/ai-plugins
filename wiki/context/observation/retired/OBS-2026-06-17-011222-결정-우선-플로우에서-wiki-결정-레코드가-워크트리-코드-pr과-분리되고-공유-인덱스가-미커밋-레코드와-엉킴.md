## 관찰

decide → define → start 순서의 결정 우선 플로우에서, wiki 결정/반려 레코드는 `task-github:start`가 격리 워크트리를 만들기 **전에** 메인 트리에 캡처된다. 이후 코드 변경은 워크트리 브랜치에 쌓이고 결정 레코드는 메인 트리에 남아, PR(브랜치 산출물)이 근거 레코드를 깔끔히 포함하지 못한다. 더해, wiki context 인덱스(`decision.md`/`rejected_decision.md`/`task.md` 등)는 모든 레코드의 공통 append 지점이라, 메인 트리에 남아있던 **다른** 미커밋 레코드(이번엔 2026-06-12 배치)와 한 인덱스 안에서 엉켜 작업별 부분 커밋을 막았다.

두 층위가 겹친다: ① 구조적 분리(결정은 메인, 코드는 워크트리 브랜치) ② 기계적 엉킴(공유 인덱스가 무관한 잔여 미커밋 레코드와 섞임).

## 근거

snapshot 리팩터 [Jeis-Jw/ai-plugins#1](https://github.com/Jeis-Jw/ai-plugins/issues/1)에서 실제 발생. DEC-2026-06-17-002727 / REJ-2026-06-17-002650을 메인 트리에 캡처 → `start`가 `issue-1` 워크트리 생성 → 코드+SSOT+문서는 브랜치/PR(#2), DEC/REJ/TASK는 메인 트리로 분리. 커밋 시 공유 context 인덱스가 2026-06-12 배치와 섞여 단일 커밋(6d53872)으로 묶을 수밖에 없었다.

## 영향

- PR이 "왜(rationale) + 무엇(code)"을 한 단위로 담지 못해 리뷰 자기완결성 저하 — 이번엔 pr-verifier에 DEC/REJ 경로를 별도 주입해 보완.
- 작업별 wiki 커밋 분리 불가 → 무관한 배치가 한 커밋에 섞임.
- 워크트리 SSOT의 `[[DEC-...]]` prose 링크가 DEC 커밋 전까지 일시적 dangling(무결성엔 영향 없음 — prose 링크는 refresh 미검증 대상).

## 현재 처리

이번엔 분리를 수용: 코드는 PR(머지), 결정 그래프는 메인에 별도 커밋. PR 본문·커밋 메시지·SSOT에 DEC ID를 명시해 추적성 확보. 4계층 분리(mechanism=코드 vs rationale=wiki/context) 관점에선 분리 자체가 틀린 건 아니다.

## 후속 분류 조건

- 컨벤션을 확정하면 → **decision 승격**: (a) "결정 레코드는 메인에 커밋, 코드는 PR이 ID로 참조"를 정본 규약으로, 또는 (b) 반대로 "결정 레코드를 워크트리/PR에 포함"으로.
- "작업 시작 전 미커밋 wiki 레코드를 먼저 커밋/정리"하는 운영 규칙이 필요하다고 판단되면 → runbook 또는 agent-operating-policy 갱신. (이번 기계적 엉킴의 직접 원인은 *잔여 미커밋 레코드*였고, 메인이 깨끗했다면 분리 커밋이 자명했다.)
