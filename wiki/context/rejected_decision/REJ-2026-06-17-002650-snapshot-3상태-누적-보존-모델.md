## 대안

snapshot staging layer를 3상태 경로 모델로 보존한다.

- 상태 폴더 3개: `snapshot/active`, `snapshot/archived`, `snapshot/promoted`.
- 기본 `snapshot save`는 **append-only** — 매 저장마다 `SNAP-<timestamp>-<slug>` 새 파일을 만든다.
- `--continues <ref>`로 후속 체크포인트를 이전 파일에 체인한다.
- `--update <ref>`를 명시할 때만 같은 파일을 제자리 갱신한다.
- 종료는 `archive`로 `archived/`에 이동 보존하고, 미래 promote 계약을 위해 `promoted/` 슬롯을 비워 둔다.

이것이 wiki-markdown 0.7.0(commit `730db3a`)에 실제 출시된 구현이다.

## 반려 사유

snapshot의 목적은 "정식 graph 승격 전 대화 맥락 체크포인트" — 본질이 **세션 컨텍스트 메모장**이다. 3상태 누적 모델은 이 목적과 어긋난다.

- **append-only 기본값이 메모장과 반대로 동작**한다. 같은 토론을 다시 저장하면 timestamp 때문에 충돌 없이 새 파일이 쌓여 `active/`가 조용히 누적된다. 메모장은 토론당 현재 상태 하나면 충분하다.
- **`archived/`가 git과 중복**이다. vault는 git 추적 대상이라 지운 스냅샷도 히스토리에 남는다. archived/는 git이 공짜로 하는 복구 보험을 폴더로 다시 구현한 잉여다.
- **`promoted/`는 writer 없는 dead 슬롯**이다. 승격은 이미 별도 `capture`/Edit으로 수행하므로 staging layer에 promote 상태가 필요 없다.
- **이력 보유는 staging의 책임이 아니다.** 토론의 진화 이력이 필요하면 record/graph가 담당한다. `--continues` 체인은 staging에 이력 책임을 잘못 넣는다.
- Living 정본 원칙([[INT-2026-05-29-104713-single-canonical-current-state]])과 어긋난다 — "하나의 현재 상태만, 이력은 record가 보유."

## 이 대안의 취지

모든 체크포인트를 감사 추적(audit trail)으로 영구 보존하고, 승격 전/후 상태를 폴더로 구분해 staging 라이프사이클을 자기완결적으로 만들려는 의도였다. 외부 작업 시스템 없이도 "이 토론이 어떤 단계인지"를 경로만으로 표현하려 했다.

## 재고 조건

- snapshot을 git 없는 vault에서도 쓰게 되어 삭제가 진짜 비가역이 되는 경우 — archived/ 같은 soft-delete 보험이 다시 필요할 수 있다.
- 한 토론의 체크포인트 이력 자체를 1급 산출물로 조회·비교해야 하는 요구가 생기는 경우 — append/continues 체인을 재고한다.
- staging에서 record로의 자동 promote 계약을 구현하기로 하면 promoted/ 상태 슬롯을 되살린다.
