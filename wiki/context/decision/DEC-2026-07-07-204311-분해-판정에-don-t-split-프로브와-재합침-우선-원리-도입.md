## 결정

define의 절단 판정을 세 겹으로 보강한다. (1) don't-split 프로브 — 절단 사유 ①(병렬 이득)의 '서로 독립적인 조각' 판정에 정직성 검사 3개를 추가한다: 형제들의 검증 명령이 사실상 동일한가 / 형제들이 같은 shared component·기반을 수정하는가("N surfaces × same system" 패턴은 공유 기반이 곧 write-set — 표면 디렉토리 분리에 속지 말 것) / 앞 조각의 작업 context를 뒤 조각이 계속 필요로 하는가. 하나라도 해당하면 독립이 아니므로 묶는다(한 리프 + phase 체크리스트). (2) 겹침 처리 우선순위 — write-set 겹침 발견 시 same-theme이면 재합침(phase화)을 먼저 검토하고, blocked_by 직렬화는 서로 다른 deliverable일 때의 차선책으로 둔다. 직렬 리프 N개 = phase N개 + 세리머니 N배이기 때문이다. quality-gates G4의 '겹침+dependency 미선언→승급'도 같은 우선순위를 따른다. (3) 강제층 — create_issue_tree.py dry-run에 siblings_maybe_phases 비차단 경고(flat_maybe_understructured의 역방향: 동일 단일 선행에만 blocked_by된 same-theme 형제 3+개, affects_paths 공통 교집합, 검증 anchor 동일 등 신호 2+개)를 추가하고, challenge review 근거 기준에 don't-split 프로브를 포함시켜 가짜 독립을 감사한다. 묶인 큰 리프의 실행 열화는 phase 운영 규약(phase별 커밋·체크포인트 코멘트·마지막 phase=full-verify·필요시 phase별 순차 워커 세션 재진입 + run-notes 승계)으로 흡수한다 — 세리머니(이슈·PR·closeout·머지엣지)는 1회로 유지된다. [[DEC-2026-07-02-224910-orchestrate-세리머니를-merge-edge-gear로-이동-분해를-payoff-원리로-재정의]]의 절단 사유 4개 프레임워크를 대체하지 않고 그 판정의 적용 실패를 막는 보강이다.

## 취지

분해 규칙이 텍스트로 존재해도 판정 순간의 착시(표면 경로 분리=독립)를 막지 못하면 소용없다. 프레임워크 교체가 아니라 판정 정직성(프로브) + 기계 경고(강제) + 묶은 뒤의 실행 위생(phase 규약)으로 '규칙 존재'와 '규칙 적용' 사이의 간극을 없앤다.

## 배경

consumer dogfood: Lightning Santa 모바일 UI 적용(#119 루트, 리프 #120–#123)을 task-github 0.18.1로 orchestrate. #120(공통 design system/app shell)은 절단 사유 ④(선행 spec 리프)로 정당한 분리였으나, #121/#122/#123(화면군별 적용)은 선언된 affects_paths가 화면 디렉토리별로 분리돼 보였을 뿐 실제 write-set이 공유 컴포넌트(Wireframe, app shell test, receive/QR 계열)에서 겹쳤다. 결과: #122 sibling merge conflict, 사실상 동일한 typecheck/test/export 검증 3회 반복, 리프 3개분 context 로딩·closeout·ledger·label 세리머니 반복. 절단 원리 텍스트가 있는 버전에서 발생했으므로 룰 부재가 아니라 판정 실패로 확정. 사용자 피드백 문서의 결론('이슈는 작업량 단위가 아니라 독립 점유·독립 완료 단위')과 기존 절단 원리를 정직 적용한 결론이 일치했다 — 올바른 구조는 #120 + 화면 적용 단일 리프(phase 3개)였다.

## 트레이드오프

재합침 우선은 병렬 벽시계 시간을 포기한다 — solo 프로파일에서 제약은 벽시계가 아니라 토큰·세리머니이고, #119에서 병렬 이득은 sibling conflict로 상쇄된 가짜 병렬이었으므로 수용한다. 묶인 큰 리프는 단일 세션 compaction 위험을 지며, 이는 phase 운영 규약(재진입 세션)으로 완화하되 세션 재로딩 비용 일부는 남는다 — 사라지는 것은 세리머니·충돌·반복 검증이고, 남는 것은 phase당 컨텍스트 로딩이다.

## 재평가 조건

phase 체크리스트가 큰 리프에서 실제로 compaction 열화를 못 막는 사례가 반복되면(phase 재진입 규약으로도 품질 저하), 재합침 우선순위를 완화하고 blocked_by 직렬화 쪽으로 되돌리는 것을 검토한다. siblings_maybe_phases 경고의 오탐(정당한 병렬 형제를 phase 후보로 지목)이 누적되면 신호 임계를 조정한다.
