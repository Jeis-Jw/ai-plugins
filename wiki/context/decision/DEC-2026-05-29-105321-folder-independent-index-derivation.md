## 결정

인덱스 파생은 폴더 단위로 독립 처리한다. Vault 전체를 재귀 탐색해 인덱스가 필요한 폴더를 발견하되, 각 `<folder>.md`가 수집하는 문서는 해당 폴더의 직속 활성 `.md` 문서뿐이다.

하위 폴더 문서는 하위 폴더의 독립 인덱스에만 포함한다. 상위 인덱스는 필요하면 하위 인덱스 링크를 노출할 수 있지만, 하위 문서 summary를 중복 수집하지 않는다.

## 취지

인덱스는 후보군을 빠르게 좁히는 필터여야 한다. 상위 인덱스가 하위 문서까지 재귀 수집하면 nested를 도입해도 상위 인덱스가 다시 비대해지고, 같은 문서가 여러 인덱스에 반복 노출된다.

"재귀 = 폴더 발견, 비재귀 = 노트 수집"으로 나누면 nested 구조와 토큰 효율을 동시에 유지할 수 있다.

## 배경

v1 수렴 과정에서 Codex 문서와 Claude 문서 사이에 인덱스 재귀 여부 차이가 있었다. 최종 결론은 폴더 독립 파생이다.

예를 들어 `ssot/auth/auth-session.md`는 `ssot/auth/auth.md` 인덱스에만 들어가고, `ssot/ssot.md`에는 중복 summary로 들어가지 않는다.

## 고려한 대안

- 상위 인덱스가 하위 문서를 모두 재귀 수집: 전체 조망은 쉽지만 인덱스 비대화와 중복 때문에 반려했다.
- 수동 인덱스 작성: 사람/AI가 누락하기 쉬워 반려했다.
- 루트 README에 전체 문서를 모두 모으기: 다시 monolithic index가 되어 반려했다.

## 트레이드오프

상위 인덱스 하나만 읽어서 모든 하위 문서를 볼 수는 없다. 대신 사용자는 상위 인덱스에서 하위 영역 링크를 따라가며 필요한 만큼 깊게 들어간다.

하위 폴더를 만들 때 인덱스 파일도 함께 관리해야 하지만, 이는 `refresh --fix index`가 안전하게 자동화할 수 있는 영역이다.

## 재평가 조건

운영 중 전체 조망 부재가 반복적인 문제로 드러나면 별도 "map" 문서를 추가할 수 있다. 다만 각 폴더 인덱스의 직속 수집 원칙은 유지한다.
