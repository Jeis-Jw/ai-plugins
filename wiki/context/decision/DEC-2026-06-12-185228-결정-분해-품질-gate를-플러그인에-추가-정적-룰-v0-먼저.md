## 결정

task-github / wiki-markdown에 결정·분해 품질 gate를 추가한다. 우선순위는 G1(`refresh --strict` / `changed-path-stale`를 보고가 아닌 hard gate로), G2(decision 기록 품질 evaluator), G3(define/분해 품질 critic), G4(major·uncertain 작업 escalation 기준).

**MVP는 LLM-judge가 아니라 정적 룰(v0)부터** 구현한다. LLM-judge는 v1에서 evidence-first · FLAG-to-human(차단 기본값 아님)으로 붙인다. gate의 역할은 사람 확인을 없애는 것이 아니라, 사람에게 올라오기 전 저품질 결정·분해를 걸러 **확인 비용을 낮추는 것**이다.

## 취지

빠른 define 후 비동기 실행으로 넘기는 운영 모델에서는 사람이 실행 루프에 없다. 나쁜 define이 충실히 잘못 구현되어 PR에서야 드러나면 재작업이 크다. 즉 비동기 핸드오프는 define 품질 기준을 오히려 **높인다**. 품질 gate가 나쁜 define을 실행 전에 차단해 사람의 인지 부담을 결정에만 집중시키고([[INT-2026-05-29-104710-ai-driven-documentation]]), 특정 에이전트의 raw 능력에 의존하지 않는 agent-neutral 안전장치를 만든다([[INT-2026-05-29-104711-plugin-agent-neutrality]]).

## 배경

운영 모델을 "대화형 결정(동기) + 정의 후 비동기 실행"으로 정리하는 과정에서, 의사결정 품질을 모델의 raw 능력에 의존하는 대신 plugin workflow가 일부 떠안게 하자는 방향이 나왔다. 현재 플러그인은 구조·스키마·무결성(`refresh --strict` 13종)·캡처 감사·반려대안 교차점검은 갖췄으나, decision·분해의 **의미적 품질**을 자동 판정하는 gate는 없다 — 그 자리를 모델 능력과 사람 확인이 떠안고 있다. 이 필요는 어떤 에이전트로 운영하든 동일하게 성립한다.

## 고려한 대안

- **LLM-judge를 MVP로 먼저 구축** → 반려([[REJ-2026-06-12-185220-gate-mvp를-llm-judge로-먼저-구축]]). in-stack judge 천장 · 판정 불안정 · prompt 의존.
- **위키 포맷 자동화를 먼저** → 후순위. 포맷은 이미 대부분 `wiki_cli`가 자동화했고, 진짜 리스크는 나쁜 define의 비동기 증폭이라 품질 gate가 우선.

## 트레이드오프

정적 룰은 결정적 · 저렴 · blind-spot 없음이지만 의미적 얕음은 못 잡는다(그건 v1 LLM-judge + 사람 확인의 몫). gate 구축은 초기 CAPEX가 있으나 재사용 자산이고 agent-neutral이라 어떤 에이전트에도, 벤더 변화에도 이득. gate가 과하면 운영 마찰 — 그래서 차단 기본값이 아니라 FLAG-to-human.

## 재평가 조건

- 정적 룰 v0가 실제 저품질 define/decision을 충분히 못 거른다고 드러나면 v1 LLM-judge 우선순위를 올린다.
- in-stack judge 천장을 낮출 방법(다양성 판사 · 정적 룰 병용 · calibration corpus)이 검증되면 LLM-judge 비중을 키운다.
- 운영 모델이 비동기 핸드오프에서 멀어져 사람이 실행 루프에 상시 복귀하면 gate 우선순위를 내린다.
