## 결정

① 원시개념 5 + producer: 미션 계약(missions/<slug>.md — KPI·예산·게이트·완료기준·자율성 한계의 정본) / 페르소나(데이터 파일: name·role·prior·requested_tools(advisory)·activation, 실행은 generic 서브에이전트에 주입; 같은 직능에 상이한 prior 복수 배치) / run / 증거(delta) / 작업장(studio/ — backlog.md는 KPI 링크 필수, board.md는 운영 정본, minutes/는 합성본+delta만·raw는 TTL, crew/는 페르소나+개인 notes) + producer(메인스레드).
② 실행 단위 run = 일감 × ritual × crew 조합의 백그라운드 1회 실행. ritual은 두 부류 — 회의형(brainstorm·standup·retro: 산출물은 문서/결정, 무제한 병렬)과 작업형(pairing: 코드+테스트, worktree 격리). 공통 I/O 계약: 입력{ritual, agenda, participants, judge, budget, stop} → 출력{synthesis, minority, delta_log[{round, changed_what, anchor, evidence, rejected_alternative}], verdict, cost, proposals}. "meeting"은 인스턴스 용어에서 폐기.
③ 에이전트 간 대화 = 브로커의 transcript 릴레이(서브에이전트 직접 채널 없음). 소집형: 매 턴 fresh 스폰 + 디스크 상태. 캐시 규칙: 공용 시스템 → transcript(append-only) → 페르소나+턴 지시 순서로 배치해 접두사 캐시 적중 확보.
④ 동시성: 미션 복수(track 단위), producer는 이벤트 드리븐 디스패처(회수→게이트→소집→보고), 동시 상한은 미션 계약에. producer 금지 2건 — 직접 산출물 제작, 역할 판단 대리 합성(converge 합성은 브로커의 generic summarizer 스텝).
⑤ 네이밍 2층: 은유 층 owner/producer/crew/critic + track/run/convene/board/minutes/casting, 계약 층(delta·anchor·evidence·dry·budget·gate)은 은유 금지. 하니스 예약어(session·agent·task) 회피.

## 취지

미션 드리븐·살아있는 상호작용·producer 비매몰이라는 취지를, 순차 파이프라인이 아닌 포트폴리오 운영 구조로 담는다.

## 배경

session-review co-design 2루프(Codex reviewer)로 수렴. r1 explore에서 개념 모델, r2 converge에서 계약 수준 구체화, 이후 owner 대화에서 동시 다중 track 운영·용어 정합(run)·캐시 배치 규칙 확정. 정본 문서 drafts/agent-team-concept.md.

## 고려한 대안

이슈트리 오케스트레이션 확장(REJ, 별도 기록) / 상주 에이전트 대화(REJ, 별도 기록) / 인스턴스 용어 meeting 유지(작업형 run과 부정합해 폐기) / session 용어(하니스 충돌로 회피) / 공용 브로커 1개+리추얼 데이터(셋째 리추얼 생길 때 재판단으로 유예).

## 트레이드오프

fresh-per-turn은 페르소나 재주입(턴당 수백 토큰, transcript 뒤라 캐시 불가)을 감수 — 미팅당 수 k로 미미. transcript O(R²) 성장은 max_rounds 3–5와 dry 폐회로 상한. 리추얼별 브로커 스크립트 2개는 중복 소지가 있으나 공통화는 반복 확인 후.

## 재평가 조건

①R>5 장기 리추얼이 흔해지면 중간 합성 압축 도입. ②requested_tools의 하니스 수준 차단(allowed_tools)은 플러그인화 시점에. ③멀티 repo 미션은 MVP 밖 — 요구가 생기면 작업장 위치 재설계. ④셋째 리추얼 추가 시 공용 브로커 공통화 판단.
