## 결정

"작업 계획/준비" 시 **위키 작업정의(task 노드)를 먼저, 연계 GitHub 이슈를 나중에** 만든다. 조율은 전부 task-github가 한다: `define`이 위키를 감지해 작업정의 task가 있으면 재사용·없으면 `capture`(또는 다른 세션의 생성을 대기)한 뒤 "진행" 확인을 받고 루트 이슈를 만들어 `relate --add-tasks`로 역링크한다. wiki-markdown은 task-github를 모른 채 작업정의 문서만 만든다(비대칭 유지). 한쪽만 설치되면 각자 단독 수행한다(wiki: 작업정의 문서, task-github: 세션 컨텍스트 기반 이슈).

## 취지

작업정의(지시)는 수행(실행 추적)의 상류다. 위키 task = 작업지시서/정의 SoT, GitHub 이슈 = 수행 뷰. 이 순서를 consumer 워크스페이스에서도 자연스럽게 강제해, "처음 보는 세션이 작업정의를 먼저 보고 수행으로 넘어가는" 흐름을 보장한다.

## 배경

기존 `define`은 이슈 먼저 → task 노드 나중(이슈 중심)이었다. 위키 4계층은 task-github→wiki 단방향 의존만 허용한다(wiki는 task-github를 모른다). 사령관이 "위키=작업정의 핸드오프(먼저), task-github=수행 이슈(나중)" 모델을 요구했다. 구현은 PR #12.

## 고려한 대안

- **wiki가 task-github를 감지해 define를 제안**: 사용자 모델에 가깝지만 wiki→task-github 역의존이 생겨 비대칭 불변식(wiki 독립성)을 깬다 → **기각**.
- **순서를 운영정책에만 두고 mechanism은 그대로**: 정책 문구는 고치되 `define`의 issue-first를 안 고치면 실제 동작이 어긋난다 → 부분 채택(정책 + define 반전 둘 다 함).
- **최종(채택)**: 조율을 전부 task-github(감지·대기·확인)에 두고 wiki는 순수 문서 생성기로 유지. `define`를 doc-first로 반전.

## 트레이드오프

- 얻음: 비대칭 불변식 보존(wiki는 github 없는 워크스페이스에서도 독립 동작), 단일 진입(`define`)만으로 작업정의 선행 보장, 4계층 정합.
- 잃음: wiki 단독 워크스페이스에선 "이슈 자동 생성"이 없다(정상 — 각자 단독). `define`가 작업정의 노드를 먼저 확보하는 단계가 추가된다(있으면 재사용으로 비용 최소화).

## 재평가 조건

- wiki/task-github 인터페이스(타입·관계·CLI) 변경으로 `relate --add-tasks` 역링크나 감지(`[ -d ./wiki ]`)가 깨질 때.
- 작업정의↔수행을 한 진입점으로 합치는 게 더 단순해질 때(예: task-github가 wiki capture를 내부 캡슐화).
