## 취지

1인 개발자가 AI 에이전트와 협업할 때, 컨텍스트 토큰은 한정 자원이다. 본문 전체를 매번 읽으면 빠르게 한계에 도달해 후속 작업이 제약된다. 위키는 "필요한 만큼만 깊이 들어가는" 압축 계층을 기본 동작으로 제공해야 한다.

압축도 계층: **인덱스(요약 모음) → header summary(한 줄) → 본문 섹션 → 본문 전체.**

## 배경

AI는 과거 결정 이유를 모르면 그럴듯하지만 틀린 결정을 반복한다. 그렇다고 매번 모든 본문을 읽으면 토큰이 폭증한다. 두 실패 모드 — *맥락 부족* vs *컨텍스트 포화* — 사이에서 균형을 잡는 것이 위키 시스템의 최우선 설계 제약이다.

따라서:
- `summary`/`tags`는 필수 검색 표면 (Stage 1).
- 본문 섹션 헤더는 **고정 계약** — Stage 2가 H2 정규식으로 의존.
- 본문 전체 Read는 명시적 요청(`--stage 3` 또는 `--read`)이 있을 때만.
