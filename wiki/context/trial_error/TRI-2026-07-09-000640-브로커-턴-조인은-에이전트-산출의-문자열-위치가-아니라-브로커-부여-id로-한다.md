## 교훈

에이전트 간 산출을 조인할 때는 **오케스트레이터(브로커)가 부여한 안정 id**를 유일한 조인 키로 써라. LLM 산출의 배열 순서·개수·자연어 표현(제목·문구)은 비결정적이므로 조인 키로 절대 신뢰하면 안 된다. 브로커가 각 항목에 id를 찍고, 다음 에이전트에게 그 id를 제시하고, 스키마에서 id 에코를 필수로 강제하고, id로 조인한다. 매칭 안 되는 id는 조작으로 간주해 무시(fabrication 차단). anchor enum 등 구조 제약은 코드에서 재확인(defence-in-depth).

## 상황

studio 브로커(Workflow 스크립트)는 여러 fresh 서브에이전트의 산출을 모아 조인한다. 두 지점에서 같은 실수: (1) brainstorm — critic의 verified[]를 제출 delta에 **배열 index**로 zip(모델이 순서 바꾸면 오귀속). (2) pairing — dev의 defended[]를 qa의 openFailures에 **제목 문자열**로 매칭(두 에이전트가 같은 실패를 다르게 표기 → 매칭 실패 → 방어해도 open에 잔존, defended·open 동시 증가 모순). (2)는 v0.1.0 baseline dogfood run에서 pairing critic이 모순을 alive=false로 Kill하며 노출됨(OBS-2026-07-08-225315).

## 피해야 할 것

critique.verified[i] ↔ submitted[i] 같은 위치 zip. dev.failure_title === qa.failure.title 같은 자연어 문자열 매칭. dev.unresolved로 '아직 열린 것'을 역추론(default-close) — 누락이 조용히 사라진다.

## 대안 또는 우회

brainstorm: TURN 제출 delta에 브로커가 id 부여 → CRITIC_SCHEMA.verified.items에 id 필수 → Map(id→verdict) 조인 (v0.1.0, 66548cd). pairing: qa 실패에 failureSeq 단조 id → DEV_SCHEMA.defended[].failure_id 필수 → openById 맵 id 조인, id로만 open 제거 (commit 15553e4, v0.1.1). default-OPEN: 명시적으로 방어된 id만 제거.

## 현재도 유효한가

유효. Workflow 하니스가 서브에이전트 직접 채널을 안 주는 한(브로커 릴레이가 필수) 모든 신규 리추얼 브로커에 적용된다. 셋째 리추얼 추가 시 이 조인 규약을 공통 브로커 계약으로 승격 후보.
