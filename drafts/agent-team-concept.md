# 살아있는 에이전트 팀 — 컨셉 초안 r1 (가칭 미정)

> 상태: session-review co-design 라운드 1 (explore). 확정안 아님.
> 이 문서는 방향(direction) 문서다 — 스키마·CLI 계약·파일포맷은 다루지 않고,
> 개념 분해와 경계만 다룬다. 반박·대안·빠진 축을 환영한다.

## 1. 취지

큰 틀의 미션을 주면, 메인스레드 + 서브에이전트 시스템이 **하나의 스타트업 팀처럼**
그 미션에 관련된 모든 영역을 스스로 수행한다 — 기획, 리서치, 디자인, 개발,
테스트, QA, 마케팅.

핵심 성질 세 가지:

- **살아있음**: 에이전트끼리 상호작용한다. 기획 에이전트끼리 브레인스토밍하고,
  dev와 QA가 만들고-깨며 짝으로 일한다. 정해진 파이프라인의 스테이지가 아니라,
  서로의 산출물에 반응하는 팀.
- **미션 드리븐**: 일감 목록을 받는 게 아니라 미션을 받는다. "일을 정의하는 행위"
  자체가 팀 안에서 일어난다.
- **메인스레드 = CEO**: 사용자와 소통하고, 회의를 소집하고, 결과를 중계하고,
  게이트를 지킨다. 작업 처리 오케스트레이션에 매몰되지 않는다.

### 반(反)목표 (non-goals)

- **이슈트리 순차 처리기의 확장이 아니다.** task-github orchestrate는 "일이 정의된
  후"의 실행 엔진이고, 이 컨셉은 "일의 정의"까지 팀 안에 들어온다. 이슈트리는
  코드 배송이 무거워지면 붙일 수 있는 실행 백엔드 옵션이지 코어가 아니다.
- **조직도 시뮬레이터가 아니다.** 사람 흉내(직급·근태·사내정치)가 아니라 협업
  구조(논쟁·짝·제안)를 시뮬레이션한다. "살아있음"은 목적이 아니라 품질 수단이다 —
  논쟁이 품질을 만들고, 제안권이 커버리지를 만든다.

## 2. 기존 자산과의 관계

| 자산 | 이 컨셉에서의 역할 | 의존 강도 |
|---|---|---|
| task-github orchestrate | 루프+서브에이전트로 단위 작업을 자율 수행하는 패턴의 **원형(참고)** | 비의존 |
| wiki-markdown | 회사 두뇌 — 결정·기각·관찰의 장기 기억 | 있으면 활용, 없으면 team/ 파일 폴백 |
| session-review | dev↔QA 페어링의 원형 (worker↔reviewer 루프) | 패턴 재사용 |
| deep-research | 리서치 역할의 도구 | 도구로 호출 |
| task-github (레일 전체) | 코드 배송 레일 | 선택적 백엔드 (§9) |

## 3. 개념 모델 — 원시개념 3 + 1

### 3.1 페르소나 (Persona)

- 역할 프롬프트가 아니라 **인격**: 이름 + 관점(prior) + 판단 성향.
- 같은 역할에 서로 다른 prior를 가진 복수 인격을 둔다 (기획A=성장 우선,
  기획B=리스크 우선). 같은 프롬프트 둘을 돌리면 서로 동의만 한다 —
  **다른 prior가 논쟁의 원료**다.
- 디스크 기억: `team/roles/<이름>/notes.md`. 세션이 죽어도 인격은 연속된다.
  fresh로 소집돼도 자기 노트 + 작업장을 읽으면 온보딩 완료.

### 3.2 미팅 (Meeting)

- 하니스 제약: 서브에이전트끼리 직접 대화할 수 없다 → **브로커**가 필요하다.
- 브로커 = 워크플로 스크립트(결정적 턴테이킹). 소규모 교환은 메인 중계도 가능.
- 미팅의 정의 = { 참석자, 안건, 턴테이킹 규칙, 종료 조건, 산출물 계약 }.
- **산출물 없는 라운드 = dry.** dry 2회면 폐회. "산출물 없는 회의는 회의가 아니다."
  여기서 산출물 = 반박, 수정안, 신규 제안 중 하나. 동의 요약은 산출물이 아니다.
- 미팅은 **백그라운드로 실행**된다 — CEO는 회의가 도는 동안에도 사용자와 대화한다.

턴테이킹 골격 (brainstorm 예):

```
transcript = 안건 + 작업장 컨텍스트
for round in 1..N:
    for 참석자 in 참석자들:
        발언 = agent(참석자.페르소나 + transcript)   # 반박/제안 강제 규범 포함
        transcript += 발언
    판정 = agent(사회자, "이번 라운드에 새 정보가 있었나? 수렴했나?")
    if 판정.dry: break
return 합성(합의안 + 소수의견)
```

### 3.3 작업장 (Workspace)

- `team/` 디렉토리 = 사무실. 에이전트 간 비동기 소통의 물리 공간.
  - `mission.md` — 미션 + KPI + 게이트 정책
  - `backlog.md` — 마크다운 백로그 (이슈트리 아님)
  - `discussions/` — 회의록. Slack 채널 느낌의 휘발성 기록
  - `roles/` — 페르소나 정의 + 개인 노트
- 굳은 지식만 wiki로 승격한다 (기존 capture 정책 그대로: 결정/기각 승격은
  사용자 확인 게이트).

### 3.4 CEO (메인스레드)

- **사용자 인터페이스**: 보고, 질문 큐, 게이트 요청.
- **소집자**: 어떤 리추얼을 언제 열지 판단.
- **중계자**: 회의 합성본 → 작업장 반영 → 다음 소집 판단.
- **게이트키퍼**: 결정·비가역·외부공개는 사용자 승인으로만 통과.
- 금지 사항: CEO가 직접 산출물을 만드는 것(팀 우회), 회의 전문을 전량
  정독하는 것(합성본만 소비 — 컨텍스트 보호).

## 4. 리추얼 카탈로그 (v0)

| 리추얼 | 구조 | 종료 조건 | 산출물 |
|---|---|---|---|
| 브레인스토밍 | diverge(N 병렬 발산) → debate(라운드 논쟁) → converge(합성) | dry 2회 or 수렴 판정 | 합의안 + 소수의견 |
| 페어링 (dev↔QA) | 구현 → 공격 → 수정 루프 | QA가 못 깸 or 예산 소진 | 코드 + 검증 리포트 |
| 스탠드업 | 각 역할이 작업장 읽고 1턴 보고 | 1라운드 고정 | 상태 요약 + 블로커 목록 |
| 회고 | 사이클 산출물 검토 | 1라운드 | wiki 승격 후보 (관찰/시행착오) |
| 데모 | 산출물을 사용자에게 미리보기 | — | Artifact / 리포트 |

리추얼을 데이터(정의 파일) + 공용 브로커 하나로 갈지, 리추얼별 스크립트로 갈지는
미결 (§10-2).

## 5. 운영 루프

- **이벤트 드리븐**이 기본: (a) 회의 종료 알림 (b) 사용자 응답 (c) 타이머 폴백 —
  이 세 가지 신호로 CEO가 깨어나 다음 소집을 판단한다. 고정 주기 사이클(스프린트)은
  강제하지 않는다. 리듬은 미션이 만들고, 타이머는 안전망이다.
- 미션 라이프사이클: `draft → active → (pivot | paused) → done`. pivot은 결정
  기록으로 남긴다.
- 상주 방식: 세션 내 상주(/loop dynamic — 사용자와 실시간 대화 가능)가 기본.
  cron 야간 가동(아침 보고)은 후순위 확장.

## 6. 자율성과 게이트

| 팀이 자율로 하는 것 | 사용자 게이트 |
|---|---|
| 회의 소집·진행 | 신규 에픽 / 방향 전환 |
| 초안 작성, 리서치, 구현+검증 | 결정·기각의 wiki 승격 |
| 작업장 갱신, 백로그 제안 | 외부 공개 (발행·배포·계정 행위) |
| 역할 노트 기록 | 예산 상향 |

- **질문 큐**: 게이트 대기로 한 레인이 막혀도 나머지 레인은 계속 돈다. CEO는
  막힌 질문들을 모아 사용자에게 배치로 묻는다.

## 7. 실패모드와 대응

1. **비싼 연극** — 서로 칭찬하며 빠르게 수렴하는 가짜 논쟁.
   → 상이한 prior 강제(3.1), "동의 요약 금지" 규범, 라운드 산출물 강제(3.2).
2. **백로그 폭주** — 그럴듯한 쓰레기 일감 무한 생성.
   → 모든 백로그 항목은 미션 KPI에 연결 강제, 에픽 단위 사용자 게이트.
3. **토큰 화재** — 상주 루프 + 다인 회의의 비용.
   → 미팅별 토큰 예산, dry 조기 폐회, 할 일 없으면 긴 sleep.
4. **에코챔버 품질** — 에이전트가 만들고 에이전트가 승인.
   → 검증 역할은 adversarial 프롬프트(반박이 임무), acceptance criteria는
   만들기 전에 고정.
5. **CEO 컨텍스트 고갈** — 모든 대화가 메인을 경유하며 누적.
   → 회의는 백그라운드, CEO는 합성본만 소비, 팀 상태는 전부 디스크에.

## 8. MVP 슬라이스

플러그인화 전에 이 repo에서 재료 4개로 실험한다:

1. 페르소나 4개 — 기획A / 기획B / dev / QA (마크다운 파일)
2. 미팅 브로커 워크플로 2개 — `brainstorm`, `pairing`
3. `team/` 작업장 스캐폴드
4. CEO 행동규약 스킬 1개 — 소집·중계·게이트 규약

검증 시나리오: 장난감 미션 1개 → 기획 브레인스토밍 → 합의안 → dev↔QA 페어링 →
데모. 풀사이클 1회.

**판정 기준: 연극인가 팀인가** — 라운드마다 실질 산출물(반박·수정·신규안)이
있었는가, 합의안이 첫 발언의 재포장이 아닌가.

## 9. 확장 방향 (MVP 이후)

- **동적 채용**: planner가 필요 역할을 제안 → 사용자 승인 → 페르소나 생성.
- 리서치 / 디자인 / 마케팅 레인 활성화 (마케팅의 외부 발행은 상시 게이트).
- **task-github 접목**: 코드 배송이 무거워지면 백로그 항목을 이슈트리로 승격,
  orchestrate에 위임.
- 멀티 미션 / 포트폴리오 운영.

## 10. 미결 논점 (co-design 안건)

1. **상주 vs 소집**: 팀원을 세션 내 상주(SendMessage로 문맥 유지, 비쌈)로 둘까,
   회의 때만 소집(fresh + 작업장 온보딩, 싸고 견고)할까. 초안 권고: 소집형.
2. **미팅 브로커 구조**: 공용 브로커 1개 + 리추얼 정의 데이터 vs 리추얼별
   워크플로 스크립트. 초안 권고: MVP는 리추얼별 스크립트 2개로 시작, 셋째
   리추얼이 생길 때 공용화 판단.
3. **미션 도메인 범위**: 코드 제품 한정인가, 비개발 미션(보고서·콘텐츠 제작)도
   1급 시민인가. 초안 권고: 산출물=파일이면 도메인 불문 — 단 MVP 검증은 코드
   미션으로.
4. **"살아있음"의 최소 조건**: 이벤트 드리븐 + 제안권(이니셔티브)으로 충분한가?
   더 필요한 요소가 있나 (예: 역할 간 자발적 1:1, 감정 상태 같은 건 배제).
5. **네이밍**: `crew`는 caveman 플러그인의 cavecrew와 충돌. 후보: `team`,
   `studio`, `company`, `troupe`.
6. **작업장 vs wiki 경계**: `discussions/` 휘발 기록을 얼마나 남기고 언제 버릴지.
   회의록 전문 보존 vs 합성본만 보존.
