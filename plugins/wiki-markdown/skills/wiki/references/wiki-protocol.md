# AI-Native Wiki Protocol

Single-file reference for the wiki plugin. Condenses the implementation contract from the shipped mechanism documents (`rules/knowledge-protocol.md` and `skills/wiki/SKILL.md`). The plugin development repo dogfoods the design rationale under `wiki/ssot/plugin-definition/`, but consumer projects do not need that vault content for the CLI contract.

## Purpose

The wiki is the project's knowledge source of truth for AI agents. It stores current design state, operational procedures, durable intents, decisions, rejected alternatives, lessons, observations, and task handoff nodes so an agent can read minimal valid context before acting.

The vault also includes a lightweight `snapshot` staging layer for unresolved conversation context. Snapshots are searchable/loadable by `snapshot` commands, but they are deliberately outside the canonical graph until a user or agent explicitly captures the distilled knowledge into a graph type.

## Vault

- Default vault path: `wiki/`.
- Root index: `wiki/README.md`.
- Folder indexes: `<folder>/<folder-name>.md` — e.g. `wiki/ssot/ssot.md`.
- `ssot/` and `runbook/` may contain nested folders. Each folder index lists only its direct child notes.
- Indexes are derived from note frontmatter `summary` (regenerated by `refresh --fix index`).
- `snapshot/active` holds discussion checkpoints outside the graph. `snapshot/snapshot.md` is an orientation/index file maintained by snapshot commands.

## Types

| Type | Folder | Form | ID format |
|------|--------|------|-----------|
| `ssot` | `ssot/` | living current truth | `<slug>.md` |
| `runbook` | `runbook/` | living procedure | `<slug>.md` |
| `intent` | `context/intent/` | record durable goal | `INT-YYYY-MM-DD-HHMMSS-<slug>` |
| `decision` | `context/decision/` | record decision | `DEC-YYYY-MM-DD-HHMMSS-<slug>` |
| `rejected_decision` | `context/rejected_decision/` | record rejected alternative | `REJ-YYYY-MM-DD-HHMMSS-<slug>` |
| `trial_error` | `context/trial_error/` | record trap or lesson | `TRI-YYYY-MM-DD-HHMMSS-<slug>` |
| `observation` | `context/observation/` | record observation not yet classifiable | `OBS-YYYY-MM-DD-HHMMSS-<slug>` |
| `task` | `task/` | **third category** — task handoff/context bridge, optionally linked to external work | `TASK-YYYY-MM-DD-HHMMSS-<slug>` |
| `snapshot` | `snapshot/active` | staging checkpoint outside the graph | `SNAP-<slug>` |

Living notes are updated in place. Records are immutable in meaning and are retired or superseded. **`task` is a third category**: its body is updated in place (living-like) and it carries relations (record-like), but its lifecycle is binary by path — active `task/` vs done `task/done/` (and `task/retired/` for an invalid task). Move it with `complete` / `reopen` (not supersede). A pure leaf: it points outward and nothing points back at it (reverse is derived backlinks). It works without any external task plugin; when an external work system is used, `relations.tasks` stores links to that system and the external plugin owns interpretation/synchronization.

**`snapshot` is not a graph type.** It has an active-only path lifecycle (`snapshot/active`) and its own CLI. It is excluded from graph `recall`, relation resolution, `refresh --strict`, and duplicate-basename checks. A `snapshot save` rewrites the file for that slug in place (preserving `created_at`, stamping `updated_at`); a new slug starts a new file. `snapshot discard <ref>` deletes an active snapshot — git retains history, so there is no archived or promoted state folder.

## Frontmatter

Every doc starts with YAML frontmatter. The CLI enforces a restricted YAML subset (see "YAML Subset" below).

### Common fields (all types)

| Field | Required | Type | Meaning |
|-------|----------|------|---------|
| `title` | yes | scalar string | Human-readable title; free to change (decoupled from the basename). |
| `created_at` | yes | ISO date `YYYY-MM-DD` | Creation date; `capture` fills it automatically. |
| `summary` | yes | one-line string | **Source of derived indexes.** Self-contained, search-friendly. Don't restate the title. |
| `tags` | yes | inline list | Controlled vocabulary; powers faceted search. |
| `audience` | optional | list | `[human, agent]` (default), or one of `[agent]` / `[human]`. |
| `search_terms` | optional | inline list | Surfaces in `recall --stage 1` matching even when summary/tags don't hit. |

### Type-restricted fields

| Field | Allowed on | Notes |
|-------|-----------|-------|
| `verified_at` | `ssot`, `runbook`, `trial_error`, `observation` | Date this note was last confirmed valid. capture rejects this field on other types (exit `2`). |
| `affects_paths` | `ssot`, `runbook`, `trial_error`, `observation` | Code paths (globs allowed: `src/auth/**`). Drives `changed-path-stale`. capture rejects this field on other types (exit `2`). |

### Lifecycle fields (top-level, records only)

**These live at the top level — not inside `relations`.**

| Field | Meaning | Set by |
|-------|---------|--------|
| `supersedes` | inline list of basenames this record *replaces* | `capture --supersedes` or `retire` |
| `superseded_by` | single basename of the *replacement* record | `retire` |
| `retired_at` | ISO date the record was retired | `retire` |
| `retired_type` | `deprecated` or `superseded` (v1: two values only) | `retire` |

### Forbidden fields

- `id` — basename is the canonical ID
- `status` — path (`retired/`) is the canonical state
- `classified_as` — v1 dropped this; observation lifecycle uses `deprecated`/`superseded` only (per the v1 §17 deferral list)
- `relations` on living types (`ssot`, `runbook`) — invariant
- Lifecycle fields nested inside `relations`

### Serialization rules (write path)

- Scalars: `key: value` (no special quoting).
- Lists: inline `key: [a, b, c]` (comma + space). An empty list omits the key entirely.
- `relations`: emit only sub-keys with non-empty values. If every sub-key is empty, omit the `relations` block.
- Key order (capture output): `title`, `created_at`, `summary`, `tags` → `verified_at`, `audience` → `search_terms`, `affects_paths` → `supersedes` → `relations`. `retired_at` / `retired_type` / `superseded_by` are appended at retire-time.

Hand-edited block-style lists and `# comments` round-trip on read.

## Relations

Canonical relations are YAML plain IDs in frontmatter — *not* body wikilinks.

| Type | Allowed sub-keys |
|------|------------------|
| `intent` | (none — it's a hub) |
| `decision` | `intents`, `rejected_decisions`, `ssot`, `tasks` |
| `rejected_decision` | `intents` |
| `trial_error` | `decisions`, `tasks` |
| `observation` | `ssot`, `runbook`, `decisions`, `tasks` |
| `task` | `intents`, `decisions`, `ssot`, `tasks` |
| `ssot`, `runbook` | (none — `relations` key forbidden) |

Each sub-key takes an inline list. Wiki refs are **always full basenames**; `tasks` alone takes external work refs (`owner/repo#N`, `github:owner/repo#N`). These refs are links, not runtime dependencies; wiki-markdown validates their shape but does not call or interpret external systems.

```yaml
relations:
  intents: [INT-2026-04-17-143052-signup-speed]
  rejected_decisions: [REJ-2026-04-17-143055-email-auth]
  tasks: [owner/repo#18]
```

Hubs (`intent`, `ssot`, `runbook`) never store backlinks; backlinks are derived at query time by scanning record frontmatter.

## Sections (§8 fixed structure)

Section headers and order are **fixed** — `recall --stage 2` relies on them. Section bodies that overflow ~500 B are truncated with a `…` marker and a "full text via `--read`" hint.

### `intent`
| Section | Intent |
|---------|--------|
| `## 취지` | The durable principle that should outlive specific decisions/implementations. |
| `## 배경` | Why this principle matters; the context that produced it. |

### `decision`
| Section | Intent |
|---------|--------|
| `## 결정` | What was decided — one clear paragraph. |
| `## 취지` | Which intent(s) this serves, and how. The *why* behind `relations.intents`. |
| `## 배경` | Constraints, data, stakeholders at decision time. Must remain auditable a year later. |
| `## 고려한 대안` | Brief alternatives without their own REJ doc. Major rejections get a `rejected_decision`. |
| `## 트레이드오프` | What was gained vs given up (weighing intents against each other). |
| `## 재평가 조건` | The conditions that would force a re-look. Forward-looking expiry triggers. |

### `rejected_decision`
| Section | Intent |
|---------|--------|
| `## 대안` | The rejected alternative's content. |
| `## 반려 사유` | Why not chosen; why the winning decision won. |
| `## 이 대안의 취지` | The intent this alternative *tried* to serve. The intent may still be live even if the alternative is dead. |
| `## 재고 조건` | What conditions would make this worth revisiting. |

### `trial_error`
| Section | Intent |
|---------|--------|
| `## 교훈` | "Next time, watch for X" — the most important one line. `refresh --check empty-lesson` flags placeholders/blank here. |
| `## 상황` | When / where this trap appeared. Reproducible. |
| `## 피해야 할 것` | Concrete anti-patterns, generalized. "Be careful" doesn't count. |
| `## 대안 또는 우회` | The correct approach / workaround / debug path. |
| `## 현재도 유효한가` | Is this still reachable today? Drives the deprecated decision. |

### `observation`
| Section | Intent |
|---------|--------|
| `## 관찰` | What was found, stated as fact. |
| `## 근거` | Files / tests / logs / phenomena it rests on. Reproducible. |
| `## 영향` | Why this could matter later (blast radius, adjacent systems). |
| `## 현재 처리` | Did the current task handle it, defer it, or work around it? |
| `## 후속 분류 조건` | Conditions that would promote this to TRI/DEC/SSOT-update/retire. |

### `ssot`
| Section | Intent |
|---------|--------|
| `## 현재 상태` | How it works **now**; no history. |
| `## 취지` | Prose `why` only for SSOTs without an upstream decision; decision-backed SSOTs rely on backlinks. |
| `## 구성요소` | Modules, resources, contracts. Wikilink to deeper SSOTs. |

### `runbook`
| Section | Intent |
|---------|--------|
| `## 목적` | For whom, in what situation, to achieve what. |
| `## 절차` | Numbered steps; each with a verifiable result. |
| `## 주의점` | Common mistakes, rollback, failure checks. |

### `task`
| Section | Intent |
|---------|--------|
| `## 개요` | What this work is, at a glance — the bridge summary a wiki reader sees. |
| `## 근거` | Which decisions / intents this work flows from, and why it exists now. |
| `## 범위와 완료 기준` | Scope boundary + what "done" means. Detailed execution lives in the linked external work item when one exists; otherwise the task itself is the handoff anchor. |

## Lifecycle

Active records live in their type folder. Retired records move to `<folder>/retired/`.

`retired_type` is exactly one of:
- `deprecated` — wrong or no longer valid.
- `superseded` — valid at the time but replaced by a newer record.

A supersession stores both sides:
- New record: top-level `supersedes: [old-id]`.
- Old record: top-level `superseded_by: new-id`, `retired_at`, `retired_type: superseded`.

Observation uses the same two-value lifecycle. When an observation eventually points to a `trial_error`, `decision`, or another observation, retire it as `superseded` with that successor as the primary replacement.

## CLI Contract

The bundled CLI (`scripts/wiki_cli.py`, stdlib only) supports nine top-level subcommands.

| Subcommand | Purpose |
|------------|---------|
| `init` | Create the vault skeleton and derived indexes (including `context/observation` and `task/` with its `done/` and `retired/`). Idempotent. |
| `capture <type>` | Create a note, resolve friendly refs to basenames, validate task refs, enforce v1 field scopes, regenerate touched indexes. `<type>` includes `task`. |
| `retire <basename>` | Retire (`--type deprecated`) or supersede (`--type superseded --superseded-by <ref>`) a context record; rewrite both sides of the supersede edge; regenerate indexes. A `task` may be retired `deprecated` only (use `complete` to finish it). |
| `complete <basename>` | Move a `task` from active (`task/`) to done (`task/done/`). Task-only. Refuses to overwrite an existing destination. |
| `reopen <basename>` | Move a done `task` back to active (`task/`). Task-only. |
| `relate <basename>` | Add relations to an existing document without hand-editing frontmatter. Task nodes may add `intents`/`decisions`/`ssot`/`tasks`; immutable records only accept external `tasks`. |
| `snapshot` | Manage discussion checkpoints outside the canonical graph. Subcommands: `save`, `list`, `search`, `load`, `discard`. |
| `recall` | Read-only query — Stage 1 (summary scan), Stage 2 (`--section <name>`), Stage 3 (full doc), batch `--read a,b,c` (preserves input order), derived backlinks via `--backlinks-of` (done tasks included by default). |
| `refresh` | Integrity report. Read-only by default; `--strict` exits `6` on issues; `--fix` accepts only `index` and `retired-in-index` (the safe, purely-derived fixes). |

### Snapshot contract

`snapshot save` requires `--title`, `--summary`, and `--tags`. It writes fixed sections for `현재 논의`, `배경`, `정해진 것`, `아직 열린 질문`, `다음에 볼 것`, `관련 파일/문서`, and `승격 후보`. The basename is `SNAP-<slug>` (slug from `--slug` or the title). Saving the same slug again rewrites that one file in place — `created_at` is preserved and `updated_at` is stamped; a new slug starts a new `SNAP-...` file. There is no append-only accumulation.

`snapshot list [query]` and `snapshot search <query>` search snapshot title, summary, tags, search terms, and body across active snapshots. `snapshot load <ref>` accepts a full snapshot basename or slug fragment. `snapshot discard <ref>` deletes an active snapshot; git retains the history, so there is no archived or promoted folder.

### Refresh checks (13 total)

| Check | Detects |
|-------|---------|
| `stale` | `verified_at` older than the per-type threshold |
| `supersede` | Both ends of a supersede edge agree |
| `broken-rel` | A `relations.*` ref points at nothing (or at an index file) |
| `task-ref` | Malformed `tasks` entries (must be a supported external work ref such as `owner/repo#N` or `github:owner/repo#N`; quoted human edits are accepted) |
| `orphan` | Record with neither backlinks nor outbound relations |
| `index` | Index file missing or out-of-sync with the folder |
| `retired-in-index` | Index still lists a retired record |
| `active-ref-retired` | Active record's `relations.*` points at a retired target |
| `tags` | `tags` field invalid (empty, non-list, etc.) |
| `changed-path-stale` | `affects_paths` changed since last `verified_at` |
| `duplicate-basename` | Two files normalize to the same NFC basename |
| `empty-lesson` | `trial_error`'s `## 교훈` section is blank/placeholder |
| `schema` | Frontmatter integrity (forbidden fields, lifecycle position, ISO date validity, placeholder values, relation-target type mismatch) |

`refresh --check all` runs all of the above. `--path <subdir>` scopes the scan to a subtree. `refresh --fix` rejects any token outside `{index, retired-in-index}` with exit `2` — including a bare `--fix`.

### Exit codes

| Code | Meaning |
|------|---------|
| `0` | Success |
| `1` | General / unexpected error |
| `2` | Argument or usage error (includes bare `--fix`, invalid `--check`, placeholder inputs) |
| `3` | Vault missing |
| `4` | Validation failure (missing or ambiguous ref, retired target, etc.) |
| `5` | Living slug already exists |
| `6` | `--strict` refresh found issues |

### Fuzzy reference resolution

- `capture --supersedes` and relation args (`--intents`, `--ssot`, …) accept *slug fragments* (e.g. `signup-speed` → `INT-2026-04-17-143052-signup-speed`) — fuzzy by default.
- List-valued relation args are repeatable: `--intents a,b --intents c --intents a` is flattened and de-duplicated in first-seen order before refs are resolved. `relate --add-*` uses the same normalization.
- `retire --superseded-by` also accepts fragments.
- `retire <basename>` (the positional target) and `recall --read` require **exact basenames** by default for safety. Opt into fragment resolution on `recall --read` via `--fuzzy`.

Ambiguous fragments and unresolved refs raise structured errors (`ref_ambiguous`, `ref_missing`) with exit `4`.

## Schema enforcement parity

`capture` and `refresh --check schema` share the same module-level validators (`_is_valid_iso_date`, `_is_placeholder_value`). What capture rejects at write time, schema flags at read time — no second-class entry path.

Capture rejects (exit `2`):
- `--verified-at` that is not strict `YYYY-MM-DD` (`2026-1-1` is rejected).
- `--title` or `--summary` matching `<...>` placeholder shape.
- Any `--tags` entry matching `<...>` placeholder shape.

`stale` and `changed-path-stale` silently skip notes with invalid `verified_at` so a single bad value isn't double-reported. `schema` reports the bad value once.

## Unicode normalization (NFC)

- All input paths (slugify input, `--slug`, relation refs) and all on-disk basenames are normalized to **NFC** at boundaries.
- Rationale: macOS APFS may return filenames as NFD (`가` decomposed to `ㄱ+ㅏ`); a NFC ref to a NFD basename would fail byte-level equality and break the resolver.
- New files the CLI creates are always NFC. Vaults imported from external tools that emit NFD filenames remain resolvable through NFC refs because `read_doc` normalizes at read time.

## YAML subset

The CLI implements a **restricted YAML subset** sufficient for frontmatter — not a general YAML parser. Deviations from full YAML:

- **Quoted strings are not unwrapped.** `summary: "value with # hash"` is stored as the literal string including the quotes.
- **Inline `#` comments are stripped.** A space before `#` ends the value. `summary: issue #42` would truncate at the space; for an intentional issue ref, attach it without a leading space (e.g. `owner/repo#42` works inside an inline list).
- **Block-style lists are accepted on read** (so hand-edited or externally-tooled frontmatter round-trips), but the CLI always *writes* lists inline.

For vaults with non-trivial YAML needs, run `refresh --check schema` regularly to catch silent parser disagreements.

## Deferred features

Automatic `snapshot promote` remains deferred. Promotion means distilling a snapshot into a first-class graph type (`observation`, `decision`, `ssot`, `runbook`, etc.) via `capture` or file edits. The snapshot is then discarded (git retains it); there is no `snapshot/promoted/` state folder.
